## 1. Executive Summary

### The Problem
Traditional auto-support systems fail because they are **Reactive**. They scan a model from the bottom up and make support decisions based only on the current layer. They see a "fingertip" and place a light support, unaware that this fingertip must eventually hold up an entire arm.

### The Solution: "Predictive Physics"
We are reversing the workflow.

1.  **Scan (Bottom-Up):** Map the geometry to identify all islands and connections.
2.  **Calculate (Top-Down):** Pour "virtual water" (Force/Weight) from the top of the model down to the bottom.

**Result:** When the "water" reaches the fingertip, we know exactly how much pressure is pushing down on it. We can then size the support perfectly based on the **Total Future Load**.

## 2. Glossary of Terms

*   **Anchor:** The lowest point of an island (a local minimum). This is the start of a print (e.g., the tip of a finger, the bottom of a sphere).
*   **Node:** A single "slice" of an island on a single layer. If a layer has two separate islands (left foot, right foot), that layer has two Nodes.
*   **Component:** A group of connected Nodes. A "finger" is a component. A "hand" is a component made of merged fingers.
*   **Static Suction:** The force required to peel a layer off the FEP film. Calculated as `Layer_Area * Resin_Viscosity_Factor`.
*   **Dynamic Suction (The Rip Force):** The force generated by a sudden increase in area. If Layer 10 is 1mm² and Layer 11 is 20mm², the suction spike is massive.
*   **Load Shedding:** The act of placing a support mid-way up a model (e.g., on a robust elbow) to "drain" the accumulated weight, so the delicate fingers below don't have to carry it.
*   **Cones Variable:** The integer result (0-10) from the user's Cones of Calibration test, used to determine the tensile strength of the resin.
*   **Athena Force Data:** Real-world peel force measurements captured from the Athena printer's internal sensors, used to validate and populate our physics constants.

## 3. Module A: The Geometry Scanner (Identification)

**Goal:** Identify what exists. Convert a 3D Mesh into a list of 2D shapes per layer.

### 3.1 The Concept (Plain English)
Imagine slicing a loaf of bread. Module A looks at every slice and identifies the holes in the bread. It doesn't know how the holes connect yet; it just records: "Slice 5 has a circle. Slice 6 has a slightly bigger circle."

It uses **Scanline Rasterization** (drawing lines through the shape) to turn complex 3D geometry into simple grid data.
It uses **RLE (Run-Length Encoding)** to compress that data so we don't run out of memory.

### 3.2 Technical Implementation

*   **Input:** 3D Mesh (STL/OBJ).

**Step 1: Slicing.** The mesh is intersected at fixed Z-intervals (e.g., 0.05mm).

**Step 2: Rasterization.**
*   Convert polygons to a binary grid.
*   **Optimization:** Store as RLE `[StartX, Length]`.

**Step 3: Component Labeling (Per Layer).**
*   On Layer N, group connected pixels into Nodes.
*   Assign each Node a temporary ID: `L_10_ID_1`, `L_10_ID_2`.

**Step 4: Metrics Calculation.** For every Node, calculate and store:
*   `Area_mm2`: The total filled pixels * pixel_size.
*   `Centroid`: X,Y center point.
*   `Bounds`: Min/Max X and Y.

**Output:** A massive array of Layers, where each Layer contains a list of isolated Nodes.

## 4. Module B: The Topology Graph (Relationships)

**Goal:** Connect the slices into a "Family Tree."

### 4.1 The Concept (Plain English)
Now we stack the slices back up. Module B looks at Slice 10 and Slice 11.
If the circle on Slice 10 overlaps with the circle on Slice 11, they are the **Same Object**.

*   **The Split (The Pants):** If one big circle on Slice 100 overlaps with two smaller circles on Slice 99, we found a "Split" (like a torso splitting into legs).
*   **The Merge (The Hand):** If two small circles on Slice 50 overlap with one big circle on Slice 51, we found a "Merge" (fingers becoming a palm).

### 4.2 Technical Implementation

*   **Input:** The Output from Module A.

**Algorithm: Overlap Detection.**
*   Compare Node A (Layer N) with Node B (Layer N-1).
*   Since we use RLE, we can use fast boolean math to check overlap.

**Graph Construction:**
*   Create a `GraphNode` object for every Component.
*   `Property Children`: List of connected nodes on Layer N-1 (Below).
*   `Property Parents`: List of connected nodes on Layer N+1 (Above).

**Junction Identification:**
*   **Anchor:** `Children.length == 0`. (This is a support point).
*   **Merge:** `Children.length > 1`. (Two things became one).
*   **Split:** `Parents.length > 1`. (One thing became two).

## 5. Module C: The Physics Accumulator (Calculation)

**Goal:** Calculate the "Bill" (Force) and decide who pays it.

### 5.1 The Concept (Plain English)
We treat the model as a system of water pipes. We start pouring water (Weight/Force) into the top of the model.

*   **Gravity:** As water flows down, it accumulates. The bottom pipe carries all the water from the top pipes.
*   **Suction:** If a pipe suddenly gets wider, it creates a vacuum. We add "Phantom Weight" to account for this pull.
*   **Load Shedding:** If we find a really strong, thick pipe (like a wrist), we can attach a support there to "drain" the water. This means the tiny pipes below (fingers) don't have to carry the heavy load from above.

### 5.2 Technical Implementation

*   **Direction:** Top-Down (Layer MAX to Layer 0).

**The Accumulator Object:** Passed from Parent to Child.
```typescript
interface LoadPacket {
    totalVolume: number;      // Actual weight of plastic above
    maxSuctionSeen: number;   // The worst peel force encountered so far
    accumulatedStress: number; // A score (0-100) of how badly support is needed
}
```

**Step 1: Calculate Local Load.**
*   `Node_Volume = Area * Layer_Height`.
*   `Node_Suction = Area * Viscosity_Constant`.
*   `Delta_Suction = (Current_Area - Child_Area) * Rip_Factor`.

**Step 2: Accumulate.**
*   `Current_Load = Incoming_LoadPacket + Node_Volume + Delta_Suction`.

**Step 3: The Load Shedding Check.**
*   Query Module D (Strategy): "Is this node robust enough to hold a support?"
*   Query Module E (Calibration): "Is the resin strong enough to hold this weight?"

**IF YES:**
*   Place a `Reinforcement_Point` here.
*   `Current_Load = 0` (We drained the weight!).

**IF NO:**
*   Pass `Current_Load` down to the Children.

### 5.3 The Global Equilibrium Check (Stability)
*Ensuring the "Primary Volume" is safely held by its children.*

Once all "Child Islands" (fingertips, earlobes) are identified and supported, we must determine if those supports are sufficient to hold the "Primary Volume" (the main body/palm/head) that connects them.

**The Concept:**
The Primary Volume inherits all the weight of its children. We prefer to rely on the existing child supports to hold the parent, but if the parent is too heavy, the child supports will snap.

**The Logic:**
1.  **Calculate Total Load:**
    *   `Total_Primary_Load` = (Weight of Primary Volume) + (Sum of Weight of All Child Volumes) + (Max Suction Forces).
2.  **Calculate Total Grip:**
    *   `Total_Child_Capacity` = Sum of the holding force (Newtons) of every support placed on the Child Islands.
3.  **The Balance Equation:**
    *   `Stability_Score = Total_Child_Capacity - Total_Primary_Load`

**The Decision:**
*   **IF `Stability_Score >= 0`:** The child supports are strong enough to hold the entire structure. **Action:** Do nothing. The Primary Volume is safe.
*   **IF `Stability_Score < 0`:** The system is unstable. **Action:** Place *new* `Reinforcement_Anchors` directly on the Primary Volume until the `Stability_Score` becomes positive.

### 5.4 The "Basin Filling" Algorithm (Volume Identification)
*Technique: Topological Watershed Transform (Inverted Gravity).*

To clearly define *what* constitutes an "Island" versus "The Model," we use a fluid-filling simulation.

**The Concept:**
We treat the model as a cave system. We invert gravity (or fill from the bottom-up) and start filling every "dead end" (local minimum) with a unique colored fluid.

**The Algorithm:**
1.  **Seed (Layer 0 upwards):** Every new polygon that appears where there was nothing below it is a **Source** (Local Minimum).
    *   Assign it a unique ID/Color (e.g., `Island_A`, `Island_B`).
2.  **Fill (Region Grow):** As we move up layers, the volume of `Island_A` accumulates.
3.  **The Event: Spillover (Merge):**
    *   When the polygon for `Island_A` touches the polygon for `Island_B` (or the Main Body), a **Merge Event** occurs.
    *   **CRITICAL STEP:** The volume of `Island_A` is now **LOCKED**.
    *   `Island_A_Final_Volume` = The amount of fluid accumulated *before* the merge.
4.  **Result:**
    *   The **Island Volume** is exactly the volume of the "cup" before it spilled over.
    *   This is the exact load that the support at the tip of `Island_A` must carry.
    *   Any volume *after* the merge belongs to the new Parent Volume.

### 5.5 The Rotational Equilibrium Check (Torque)
*Handling "The Leaning Tower" problem.*

Vertical strength is not enough. A model that leans heavily to one side (like a character bending forward) generates massive **Torque** that can peel the supports off the build plate even if they are vertically strong enough.

**The Logic:**
1.  **Calculate Center of Mass (CoM):** Find the weighted center of the Primary Volume.
2.  **Calculate Center of Support (CoS):** Find the average position of the existing Child Supports.
3.  **Measure the Lever Arm:**
    *   `Lever_Arm_Distance` = Horizontal distance (XY plane) between CoM and CoS.
4.  **Calculate Torque:**
    *   `Torque_Load = Total_Primary_Weight * Lever_Arm_Distance`

**The Decision:**
*   **IF `Torque_Load > Safe_Torque_Limit`:** The model is at risk of rotation/peeling.
*   **Action:** We must place a new **Reinforcement Anchor** specifically positioned to counteract the lever arm (e.g., far out under the chin or elbow), maximizing the distance from the pivot point.

### 5.6 Support Contribution Efficiency (The "Arm vs. Leg" Factor)
*Preventing "Distant Supports" from claiming they are helping.*

A support under the Hand does *technically* help hold up the Head, but far less efficiently than a support under the Neck. If we simply sum the total capacity, the math might mistakenly think the Head is safe because the Hands are well-supported.

**The Concept:**
We apply an **Efficiency Penalty** based on the horizontal distance from the Primary Volume's Center of Mass (CoM).

**The Logic:**
`Distance_Penalty` = Distance(Support_Position_XY, CoM_XY)

**The Formula:**
`Effective_Capacity = Rated_Capacity * (1 / (1 + Distance_Factor * Distance_Penalty))`

*   **Result:**
    *   **Supports under Feet (Near CoM):** Efficiency ~100%.
    *   **Supports under Hands (Far from CoM):** Efficiency ~10-20%.

**Application:**
When calculating the **Global Equilibrium (Section 5.3)**, we use the `Effective_Capacity`, NOT the raw capacity. This forces the system to realize the Head/Torso is "unsupported" even if the Hands are rock solid, triggering the placement of new supports closer to the core.

## 6. Module D: The Strategy Engine (Placement)

**Goal:** The "Artist." Decides where to shed the load to avoid scarring the face.

### 6.1 The Concept (Human Recognition)
Module C screams "I need to dump this weight NOW!" Module D looks at the model and says, "Okay, but not on the face. Put it under the chin, or on the back of the head."

**The Core Philosophy:** The human brain is hard-wired to recognize faces and creatures. A small scar on a back muscle is invisible to the human eye, but a small scar on a cheekbone ruins the entire model. Module D exists to enforce this human preference over pure mathematical efficiency.

### 6.2 Technical Implementation (The Penalty System)
Since we cannot use AI to "see" a face, we use a Weighted Cost Function based on User Vectors.

*   **Input:** User-defined Vectors (`Up_Vector`, `Front_Vector`).

**The Scar Map:** We assign a "Cost" to every potential support location.
*   **Face (Front Vector):** Cost = 1000 points (Prohibitively expensive).
*   **Side/Neutral:** Cost = 50 points.
*   **Back/Bottom (Hidden):** Cost = 1 point (Cheap).

**Scoring Logic:**
```javascript
function getScarPenalty(normal, frontVector, bottomVector) {
    // If normal points towards the Face Vector (Dot Product > 0.5)
    if (dotProduct(normal, frontVector) > 0.5) return 1000; 

    // If normal points towards the Bottom/Back (Dot Product > 0.8)
    if (dotProduct(normal, bottomVector) > 0.8) return 1;    

    return 50; // Neutral side.
}
```

**Resolution:** When Module C requests a Load Shed, Module D searches the local neighborhood for the pixel with the lowest Cost. It will naturally gravitate towards the back of the head or underside of the chin to save "money" (avoid penalties).

## 7. Module E: Calibration Logic & Athena Data

**Goal:** The "Chemist." Translates calibration data into precise math constants.

### 7.1 The Athena Integration (Hard Data)
We have access to the Athena Printer Force Sensors. This gives us a massive advantage: we don't have to guess the suction forces; we can record them.

**The Workflow:**
1.  We run test prints on the Athena (e.g., cylinders of varying diameters, cones of varying angles).
2.  The sensor records the exact Newton force required to peel each layer.
3.  We populate our Lookup Table with this hard data.

**Result:** When the software sees a 10mm² circle, it knows exactly how much force it generates because the Athena measured it.

### 7.2 The Cones Integration (User Variable)
For users who do not have an Athena printer, we use the "Cones of Calibration" as a proxy for the Athena data.

*   **Input:** Integer `Cone_Score` (User input).

**The Lookup Table (JSON - Validated by Athena):**
```json
{
  "cone_3": { "tensile_modulus": 0.8, "tip_diameter_modifier": 1.4 },
  "cone_5": { "tensile_modulus": 1.0, "tip_diameter_modifier": 1.0 },
  "cone_8": { "tensile_modulus": 1.2, "tip_diameter_modifier": 0.8 }
}
```

**Usage:**
*   Module C uses `tensile_modulus` to determine if a node will snap under load.
*   The Generator uses `tip_diameter_modifier` to set the final cylinder size.

## 8. Workflow Example: Tracing "The Arm"

Let's trace the logic for a generic arm hanging down.

**Scanning (Module A & B):**
*   The system sees one big component at the top (Shoulder).
*   It splits into the Arm.
*   It merges into the Palm.
*   It splits into 5 Fingers.
*   It ends at 5 Fingertips (Anchors).

**Accumulation (Module C - Top Down):**
1.  **Shoulder:** Volume starts accumulating.
2.  **Elbow:** Volume is high.
    *   *Check Module D:* Is the Elbow robust? Yes. Is it the Face? No.
    *   *Action:* Place a `Reinforcement_Support` on the back of the elbow.
    *   *Result:* The accumulated weight is reset to zero.
3.  **Forearm:** Volume starts accumulating again (but only from the elbow down).
4.  **Wrist:** Volume is medium.
    *   *Check Module D:* Wrist is robust.
    *   *Action:* Place `Reinforcement_Support` on the wrist.
    *   *Result:* Weight reset to zero.
5.  **Palm:** Low weight. No support needed yet.
6.  **Fingers:** Very low weight.
7.  **Fingertips (Anchors):**
    *   The system reaches the bottom.
    *   *Calculation:* "I only need to support the weight of the finger itself, because the arm and wrist were supported above."
    *   *Action:* Place `Light_Anchor_Supports` on the fingertips.

**Result:**
*   Fingertips have tiny supports (no scars).
*   Wrist/Elbow have heavy supports (holding the weight).
*   The print succeeds because we respected the physics.

## 9. Open Questions & Implementation Details (Restored)

*These questions must be answered during the development of the prototype.*

**Robustness Threshold:** How do we specifically define "Robust enough for a support"?
*   *Option A:* Pure Surface Area (e.g., >5mm²).
*   *Option B:* Surface Area + Normal Vector (Angle of the dangling bits).
*   *Recommendation:* Use both. A large area that is steeply angled might still be too weak.

**The "Check-in" Interval:**
*   Do we check for load shedding every layer? (Computationally expensive).
*   Or every 10 layers? (Faster, but might miss a sweet spot).
*   *Proposal:* Check every layer, but only trigger logic if `Accumulated_Load > Limit`.

**Load Distribution at Splits:**
*   When the Arm (100g) splits into 5 fingers, how do we divide the load?
*   *Equal Split:* Each finger takes 20g.
*   *Area-Based Split:* The Thumb (thicker) takes 40g, the Pinky takes 10g.
*   *Current Plan:* Use Area-Based splitting for higher accuracy.

**Islands vs Reinforcements:**
*   **Foundation Supports:** Placed at the very bottom (Islands/Fingers). Critical for starting the print.
*   **Reinforcement Supports:** Placed mid-air on the model to catch heavy loads. Critical for completing the print.

## 10. Future Considerations

*   **The "Sphere" Math:** We need to finalize the formula for `Delta_Area / Delta_Height`. The Sphere is the worst-case scenario for suction. We may need a specific "Sphere Detector" that identifies high-curvature zones.
*   **Variable Layer Height:** If the user prints with variable layer heights (e.g., 0.05mm for detail, 0.1mm for infill), the physics engine needs to adjust the `Node_Volume` calculation accordingly.
*   **Hollowed Models:** If a model is hollowed, the volume calculation must account for the inner void. Module A (Scanner) needs to support "Holes" inside the boolean mask.
