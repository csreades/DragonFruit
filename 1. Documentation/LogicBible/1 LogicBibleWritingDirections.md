# Logic Bible — Purpose and Guide

The Logic Bible is our place to write down how we think and work. It’s only about the Logic Bible itself: what we write here and how we write it.

## Goals

- **Capture decisions** so we don’t lose them.
- **Explain the why** in clear, everyday language.
- **Record gotchas** to prevent repeat mistakes.
- **Make repeat tasks easy** with short checklists.
- **Keep terms consistent** across all Logic Bible pages.

## Audience

- **AI agents** joining the work.
- **You (future self)** when context has faded.
- **Collaborators** who need shared understanding.

## Scope — what belongs here

- **Names and definitions** for key ideas (what we call things and why).
- **How‑tos/playbooks** for tasks we do often.
- **Gotchas, constraints, and trade‑offs** we’ve discovered.
- **Simple rules and standards** used in our logic.
- **Rationales** when a choice isn’t obvious.

## How to use

- **Skim first** to learn the language and rules.
- **Search** this folder to find topics; don’t rely on file lists.
- **Add notes** when you learn something or hit a snag.
- **Keep pages small**; link to related pages instead of making one giant page.

## How we write (style)

- **Plain language.** Prefer simple words over jargon.
- **One idea per page.** Use clear headings and short sections.
- **Do first, explain second.** Start with steps, then add background.
- **Show examples.** Use small examples or quick sketches when helpful.
- **Be specific.** Name inputs, outputs, and constraints.
- **Keep it current.** Update pages when reality changes.

## Linking and terms

- **Terms:** Use `[[Term]]` when a word has a defined meaning here. Link it to the page that defines it.
- **Relative links:** Link to other Logic Bible pages with relative paths.
- **Avoid file lists:** Don’t maintain inventories in this README; they go stale.

## Conventions

- **Z points up** (when relevant to a topic).
- **Units:** millimeters; layer height entered in microns (when relevant).
- **Keep docs accurate.** If a page is wrong, fix it or mark it clearly.
- **Link simply.** Prefer short, plain links.

## Change policy

- Prefer small edits and links over big rewrites.
- When changing a definition, add a short note with date and why.
- If something is outdated, add a brief "Deprecated" note before removal.

## How to add or update a page

- **Add a new page:** Create a small, focused Markdown file named after the idea.
- **Start with a summary:** 1–3 sentences on what the page covers and why it matters.
- **Include a quick-start:** Bullet steps for the most common task.
- **Add details:** Definitions, constraints, trade‑offs, examples.
- **Cross‑link:** Add links to related pages using `[[Term]]` or relative links.

## Out of scope

- App code, APIs, or setup instructions.
- Project or product READMEs.
- Release notes and marketing copy.
This README is only about the Logic Bible and how we write inside it.
