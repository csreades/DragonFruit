# Island Detection Method

## Overview

This document explains how the STL slicer detects and tracks unsupported regions (islands) in 3D models. The system identifies areas that would fail to print properly without support structures by analyzing the model layer-by-layer.

---

## What is an Island?

**An island is the entire 3D volume of model material that sits on unsupported regions.**

Think of it like a tree growing from a tiny seed:
- The seed (unsupported base) triggers the island detection
- The entire tree above it (all connected geometry) is part of that island
- The island continues upward until it merges with another island or reaches the top of the model

### Key Concept

Islands are **NOT** just the unsupported pixels at a single layer. They are **3D volumes** that:
1. Start when unsupported geometry is detected
2. Include all connected model material vertically above the unsupported base
3. Continue through multiple layers as the geometry grows, shrinks, or changes shape
4. End when they merge with another island or the scan completes

### Example: Cone Tip

A cone starting from a mathematical point:
- **Layer 0**: Single pixel (unsupported) → Island detected
- **Layer 1-50**: Cone grows larger → Same island continues
- **Entire cone** = One island (not 50 separate islands)

## How It Works (Simple Explanation)

### Step 1: Slice the Model

The model is divided into horizontal layers (like slicing bread). Each layer represents a thin cross-section of the model at a specific height.

**Default layer height**: 50 microns (0.05mm)

### Step 2: Convert to Pixels

Each layer's cross-section is converted into a grid of pixels (like a black-and-white image):
- **Black pixels** = Empty space
- **White pixels** = Solid model material

**Default pixel size**: 0.10mm (100 microns)

This creates a 2D bitmap representation of each layer.

### Step 3: Find Unsupported Areas

For each layer, the system compares it to the layer below:

1. **Expand the previous layer** by a "support buffer" distance (default: 0.6mm)
   - This simulates how far material can bridge without support
   
2. **Compare current layer to expanded previous layer**
   - Pixels in the current layer that DON'T overlap with the expanded previous layer are **unsupported**
   
3. **Group connected unsupported pixels**
   - Adjacent unsupported pixels are grouped into components (potential islands)

### Step 4: Track Islands Across Layers

The system follows islands as they grow upward through the model:

- **New island**: Unsupported pixels with no connection to previous layer
- **Continuation**: Pixels that overlap with an existing island from below
- **Merge**: When multiple islands join together into one

Each island gets a unique ID that stays with it throughout its lifetime.

### Step 5: Handle Merges Intelligently

When islands merge, the system needs to determine which one is the "parent":

1. **Wait 30 layers** after the merge
2. **Track which original island** the merged geometry overlaps with most
3. **Assign parent** based on maximum overlap
4. Mark other islands as "children" of the parent

This ensures accurate parent-child relationships even for complex geometries.

### Step 6: Calculate Volumes

For each island, the system calculates:
- **Volume**: Sum of (area × layer height) for all layers
- **Maximum cross-sectional area**: Largest area across all layers
- **Layer span**: First and last layer where the island appears

### Step 7: Filter Small Islands

Islands smaller than a minimum threshold are removed:
- **Default threshold**: 0.01 mm² maximum cross-sectional area
- This filters out noise and tiny features that don't need support

## Technical Details

### Coordinate System

- **Z-axis**: Vertical (height) - points upward
- **Y-axis**: Forward/backward (depth)
- **X-axis**: Side-to-side (width)

Layers are sliced at Z heights, creating horizontal XY plane cross-sections.

### Rasterization Process

**Input**: Closed polygon loops from mesh cross-section
**Output**: Binary pixel grid (Mask), then an RLE (Run Length Encoded) representation

Conceptually, rasterization still works like this:
1. Compute the bounding box of all loops.
2. Create a grid with the specified pixel size (`px_mm`).
3. For each pixel, decide whether its center lies inside any polygon.
4. Mark inside pixels as 1 (solid) and outside pixels as 0 (empty).

In practice, the implementation immediately converts these dense masks into **RLE form**:

- Each scanline row is stored as runs of `(start, length)` rather than individual pixels.
- All downstream operations (dilation, subtraction, component labeling, island tracking) operate on this RLE data instead of raw bitmaps.

**Point-in-polygon test**: Uses a standard ray‑casting algorithm with a horizontal ray and counts intersections with polygon edges.

### Morphological Dilation

To find supported regions, the previous layer is expanded (dilated) by the support buffer distance.

Conceptually, we perform a **circular dilation**:

- Any pixel within radius `support_buffer_mm / px_mm` of a solid pixel in the previous layer is considered "reachable".
- If the current layer’s solid pixel lies inside that dilated region, it is treated as **supported**.

Implementation detail:

- Dilation is performed on **RLE masks**, not on individual pixels.
- An optimized RLE algorithm (`rleIntersectDilated`) handles expanding runs and intersecting them with the current layer.

**Purpose**: Simulates material bridging capability – the current layer is "supported" if it lies within the buffer distance of material from the layer below.

### Component Labeling

Identifies connected regions of unsupported pixels.

Conceptually, this is equivalent to a flood‑fill:

- Scan the grid for unlabeled unsupported pixels.
- Grow regions to all connected neighbors (4‑ or 8‑connectivity).
- Track component ID, pixel count, and area.

Implementation detail:

- Component labeling is performed directly on **RLE masks** using an RLE‑aware connected components algorithm (`rleLabelComponents`).
- This avoids materializing full dense grids while still preserving connectivity and component metadata.

**Connectivity options**:
- **4‑connect**: Only horizontal/vertical neighbors (default).
- **8‑connect**: Includes diagonal neighbors.

### Island Tracking (IslandTracker)

Manages stable island IDs across layers and parent–child relationships.

At a high level, the tracker:

- Treats each **solid component** in a layer as a candidate chunk of 3D volume.
- Looks at how that chunk overlaps islands from the **previous layer**.
- Decides whether this chunk is a **new island**, a **continuation**, or part of a **merge**.
- Accumulates per‑layer area and volume statistics for each island.

**Processing logic per layer (current behavior)**:

1. **Find solid components** in the current layer
   - Run connected components on the **full solid mask** (RLE) for this layer.
   - Each solid component is a contiguous patch of material, regardless of whether it is supported or unsupported.

2. **For each solid component, check overlap with previous islands**
   - Compare the component’s pixels against the **previous layer’s island label grid** (also RLE).
   - Compute two sets:
     - `prevIds`: all island IDs (active or not) that this component overlaps when looking one layer below.
     - `activePrevIds`: the subset of `prevIds` whose islands are currently marked as `active`.

3. **Decide how this component should be labeled**

   - **Multiple active overlaps** (`activePrevIds` has more than one ID):
     - Treat as a **merge event**.
     - Mark the merging islands as complete at the previous layer.
     - Create a temporary **merged placeholder** island and record a `PendingMerge` entry for later evaluation.

   - **Single active overlap** (`activePrevIds` has exactly one ID):
     - Treat as a straightforward **continuation** of that island.
     - Assign the same island ID to this component and update its per‑layer area, `lastLayer`, `maxAreaMm2`, etc.

   - **No active overlaps, but overlaps with non‑active parents/placeholders** (`activePrevIds` is empty, `prevIds` is not):
     - This situation typically arises just above a merge or where a parent has been marked complete.
     - Instead of creating a new island, the tracker:
       - Follows `parentId` links from one of the `prevIds` to find the **ultimate ancestor island**.
       - Treats the current component as a **continuation of that ancestor**.
     - This is what prevents "top caps" that sit directly on existing solid from being mis‑classified as separate islands.

   - **No overlaps at all** (`prevIds` is empty):
     - This is a **true new unsupported seed**.
     - Create a new island, starting its lifetime at this layer.

4. **Assign labels for the current layer**
   - Once each solid component has an island ID, the tracker writes those IDs into a per‑layer **island label grid** (RLE rows of `(start, length, islandId)`).
   - This label grid is what downstream tools use for voxel visualization and overlay placement.

5. **Track pending merges over time**
   - For each `PendingMerge`, the tracker:
     - Records per‑candidate overlap counts over a 30‑layer evaluation window.
     - After the window, chooses the true **parent island** as the one with maximum continuation overlap.
     - Marks the other candidates as **children** of that parent.
     - Assigns the temporary merged placeholder to the chosen parent and removes placeholders from the final island list.

### Cross-Section Caching

To optimize performance, cross-sections are cached:
- Each layer's cross-section is computed once
- Reused as "previous layer" for next layer
- Cache keeps only last 3 layers to avoid memory bloat
- Reduces `computeLoopsAtZ` calls by ~50%

### Parallel Processing

Workers scan layers in parallel:
- Uses Web Workers (one per CPU core)
- Each worker processes layers independently
- Results collected and processed sequentially by IslandTracker
- Typical speedup: 2-4x depending on CPU cores

**Worker responsibilities**:
- Rasterize current and previous layer loops
- Compute supported regions (dilation + comparison)
- Label unsupported components
- Return masks, labels, and component metadata

**Main thread responsibilities**:
- Coordinate workers
- Sequential island tracking (must be sequential for stable IDs)
- Volume calculation
- Filtering and cleanup

### Volume Calculation

Volume is calculated from per‑layer areas:

```
Volume = Σ (area_at_layer × layer_height)
```

**Why use perLayerAreaMm2**:

- Areas are recorded at the time each layer is processed.
- When merges and parent reassignments happen later, the area contributions can still be attributed correctly to the parent island.
- This avoids double‑counting or losing volume during placeholder resolution.

**Alternative approaches NOT used**:

- Counting pixels only in the final label grids (can be misleading after merges and filtering).
- Full voxel counting (would be much more memory‑intensive).

### Filtering Strategy

**Post-scan volume-based filtering**:
- ALL islands tracked during scan (no size checks during detection)
- After scan completes, filter by maximum cross-sectional area
- Remove filtered island IDs from pixel labels

**Why filter by area instead of volume**:
- Area is more relevant for 3D printing
- Small base area = won't print reliably, regardless of volume
- Handles growing features (cone tips) correctly

**Default threshold**: 0.01 mm² (0.1mm × 0.1mm square)

### Placeholder Resolution

Merged islands create temporary placeholders during evaluation:

- Placeholders are marked with a flag (e.g. `isMergedPlaceholder: true`).
- A `PendingMerge` record tracks which original islands merged and how much each continues afterward.
- After evaluation, each placeholder is assigned to a **true parent** island.
- Pixel labels and island relationships are updated so that only non‑placeholder IDs remain in the final results.

**Resolution algorithm (conceptual)**:

- Build a mapping from each placeholder island ID to its chosen parent island ID.
- For each island ID that refers to a placeholder, follow the chain of parent links until reaching a non‑placeholder ancestor (the **true parent**).
- Update island relationships (`parentId`, `childIds`) so children point at real parents, not placeholders.
- Clean up any placeholder islands from the public `islands` list, while still keeping their area/volume contributions merged into the parent.

## Configuration Parameters

### Layer Height
- **Default**: 50 microns (0.05mm)
- **Range**: 10-200 microns typical
- **Effect**: Smaller = more accurate, slower scan

### Pixel Size (px_mm)
- **Default**: 0.10mm (100 microns)
- **Range**: 0.05-0.50mm typical
- **Effect**: Smaller = more accurate, more memory, slower

### Support Buffer
- **Default**: 0.6mm
- **Range**: 0.1-2.0mm typical
- **Effect**: Larger = fewer islands detected (more bridging allowed)

### Connectivity
- **Default**: 4-connect
- **Options**: 4-connect or 8-connect
- **Effect**: 8-connect finds fewer, larger components

### Minimum Island Area
- **Default**: 0.01 mm²
- **Range**: 0.0001-10 mm²
- **Effect**: Larger = fewer small islands shown

## Performance Characteristics

### Typical Scan Times
- **Small model** (20mm cube): 1-2 seconds
- **Medium model** (100mm tall): 5-10 seconds
- **Large model** (200mm tall): 15-30 seconds

### Memory Usage
- **Grid size**: width × height × num_layers × 4 bytes
- **Example**: 1000×1000 grid, 2000 layers = 8 GB
- **Optimization**: Cross-section caching keeps only 3 layers

### Bottlenecks
1. **Cross-section computation**: Slicing mesh at each Z height
2. **Rasterization**: Point-in-polygon tests for each pixel
3. **Component labeling**: Flood-fill operations

### Optimization Strategies
- Parallel worker processing (CPU cores)
- Cross-section caching (50% reduction)
- Efficient flood-fill with pre-allocated queues
- Typed arrays (Int32Array, Uint8Array) for performance

---

## Algorithm Complexity

- **Rasterization**: O(W × H × L) where W=width, H=height, L=loops
- **Dilation**: O(W × H × R²) where R=radius in pixels
- **Component labeling**: O(W × H) per layer
- **Island tracking**: O(N × M) where N=components, M=previous islands
- **Overall**: O(layers × width × height) - linear in grid size

## Edge Cases Handled

### Cone Tips (Growing from Point)
- Start with 0-1 pixels (mathematical point)
- Grow larger in subsequent layers
- **Solution**: No per-layer filtering, only post-scan filtering by total volume

### Merging Islands
- Multiple islands join into one
- **Solution**: Pending merge evaluation over 30 layers to determine true parent

### Splitting Islands
- One island splits into multiple
- **Solution**: Treated as parent completing, new children starting

### Floating Islands
- Islands with no connection to previous layer after layer 0
- **Solution**: Detected as new islands, tracked independently

### Thin Bridges
- Material that barely connects to layer below
- **Solution**: Support buffer determines if connection is sufficient

## Output Data

### ScanResults Object

Conceptually, the scan produces a `ScanResults` object with:

- **`grid`** – grid dimensions, pixel size, and world‑space origin.
- **`layers`** – per‑layer scan data (solid masks, unsupported components, counts).
- **`islands`** – the final list of tracked islands with IDs, layer span, volume, and hierarchy.
- **`islandLabelsPerLayer`** – per‑layer **RLE label grids**; each row stores `(start, length, islandId)` runs.
- **`firstHit` / `lastHit`** – for each pixel in the grid, the first/last layer where solid material appears.
- **`baseFootprint` / `baseLabels` / `compBase` / `compTop`** – helper arrays that summarize base regions and their vertical extent.

These structures together allow other systems (voxel visualization, overlays, future auto‑support) to map from abstract island IDs back to concrete positions on the model.

### Island Data

Each island entry in `islands` includes, at minimum:

- A unique stable ID.
- First and last layer where it appears.
- Status flags (active/complete/placeholder during scan; placeholders removed from final exposed list).
- Per‑layer areas (`perLayerAreaMm2`) and aggregate totals.
- Calculated volume and maximum cross‑sectional area (`maxAreaMm2`).
- Parent/child relationships (`parentId`, `childIds`) reflecting merges.

### Pixel Labels

- For each layer, `islandLabelsPerLayer` encodes which pixels belong to which island IDs (0 = not island).
- These labels are used for:
  - 3D voxel visualization.
  - Blob/overlay placement.
  - Geometric queries (e.g., fetching all pixels for an island at a given layer).
