# Island Volume Calculation and Parent-Child Hierarchy

## Overview
This document explains how the STL slicer calculates individual island volumes and determines parent-child relationships when islands merge during layer-by-layer scanning.

---

## Simple Explanation

### What is an Island?
An **island** is a region of your 3D model that doesn't have enough support from the layer below it. Think of it like a part of your print that's "floating in the air" and needs support structures.

### The Challenge
When you slice a 3D model layer by layer from bottom to top, islands can:
1. **Start** - A new unsupported region appears
2. **Continue** - An island grows or shrinks but stays separate
3. **Merge** - Two or more islands touch and become one structure

The challenge is: **When islands merge, which one should be considered the "parent" (main structure) and which should be the "child" (smaller branch)?**

### Why This Matters
- **Volume Calculation**: Each island needs its own accurate volume measurement for support planning
- **Hierarchy**: Understanding which islands merged into which helps you see the structure of your model
- **User Interface**: You want to see the 3 original islands, not confusing temporary merged structures

### Our Solution: The 30-Layer Look-Ahead

When two islands merge, we don't immediately decide which is the parent. Instead:

1. **Mark both islands as complete** at the layer where they merge
2. **Create a temporary placeholder** to represent the merged structure
3. **Wait 30 layers** and track which original island has more overlap with the continuing structure
4. **After 30 layers**, whichever island had more overlap becomes the parent
5. **Filter out the placeholder** so you only see the real islands

**Example:**
- Island #2 (medium cone) and Island #3 (small cone) merge at layer 175
- We create placeholder #4 and track overlaps for 30 layers
- After 30 layers: #2 had 177,300 overlapping pixels, #3 had 137,073
- **Result**: #2 is the parent, #3 is the child

This approach works even when islands have identical cross-sections at the merge point!

## Technical Explanation

### Volume Calculation

#### Problem Statement
When islands merge, the merged structure continues to grow. If we naively count all pixels labeled with an island ID, parent islands would accumulate the volume of their merged children, leading to inflated volume calculations.

#### Solution: Per-Layer Area Maps
Each island maintains a `perLayerAreaMm2` map that records the cross-sectional area at each layer **at the time that layer was processed**, before any relabeling from merges.

**Algorithm:**
```typescript
for (const island of islands) {
  let volumeMm3 = 0;
  
  // Sum area × height for each layer the island existed
  for (const [layer, areaMm2] of island.perLayerAreaMm2) {
    volumeMm3 += areaMm2 * layerHeightMm;
  }
  
  island.volumeMm3 = volumeMm3;
}
```

**Key Insight:** By using the pre-merge area data, each island's volume represents only its individual contribution, not the accumulated merged structure.

### Parent-Child Hierarchy with Delayed Evaluation

#### Problem Statement
When islands with identical or very similar cross-sections merge, we cannot determine the true parent based solely on volume or area at the merge point. We need to look at subsequent layers to see which island "continues" more strongly.

#### Solution: 30-Layer Evaluation Window

**Phase 1: Merge Detection (Layer N)**
1. Detect that multiple islands overlap in current layer
2. Resolve any merged placeholders to their true parents
3. Mark all merging islands as `complete` with `lastLayer = N - 1`
4. Create a new island as a merged placeholder with `isMergedPlaceholder = true`
5. Store pre-merge labels to track which candidate each pixel belonged to
6. Create a `PendingMerge` structure to track the evaluation

**Phase 2: Overlap Tracking (Layers N+1 to N+30)**
For each layer in the evaluation window:
1. When processing components, check if they overlap with the merged placeholder
2. For each overlapping pixel, look up which original candidate it belonged to using pre-merge labels
3. Increment the overlap count for that candidate

**Phase 3: Parent Determination (Layer N+30)**
1. Compare overlap counts for all candidates
2. The candidate with the most overlapping pixels becomes the parent
3. Update parent-child relationships:
   - Non-parent candidates: `parentId = parentId`
   - Merged placeholder: `parentId = parentId`
   - Parent: Add all candidates and placeholder to `childIds`

**Phase 4: Filtering**
Remove merged placeholders from final results:
```typescript
const realIslands = islands.filter(island => !island.isMergedPlaceholder);
```

#### Data Structures

**PendingMerge:**
```typescript
type PendingMerge = {
  mergeLayer: number;                    // Layer where merge occurred
  candidateIds: number[];                // Original islands that merged
  mergedIslandId: number;                // Temporary placeholder ID
  overlapCounts: Map<number, number>;    // Pixel count per candidate
  preMergeLabels: Int32Array;            // Labels before merge for tracking
  width: number;                         // Grid dimensions
  height: number;
};
```

**Island:**
```typescript
interface Island {
  id: number;
  firstLayer: number;
  lastLayer: number;
  status: 'active' | 'complete';
  perLayerAreaMm2: Map<number, number>;  // Area at each layer (pre-merge)
  volumeMm3?: number;                     // Calculated volume
  parentId?: number;                      // Parent island ID
  childIds: number[];                     // Child island IDs
  isMergedPlaceholder?: boolean;          // True for temporary merged islands
}
```

#### Placeholder Resolution
When a merged placeholder is involved in a subsequent merge, it must be resolved to its true parent:

```typescript
const resolvedIds = new Set<number>();
for (const id of prevIds) {
  const island = this.islands.get(id);
  if (island?.isMergedPlaceholder && island.parentId !== undefined) {
    // Resolve placeholder to its true parent
    resolvedIds.add(island.parentId);
  } else {
    resolvedIds.add(id);
  }
}
```

This ensures that parent-child relationships are based on the original real islands, not temporary placeholders.

### Example Walkthrough

**Model:** Three cones stacked vertically
- Cone #1: Layers 0-249 (largest, bottom)
- Cone #2: Layers 2-174 (medium, middle)
- Cone #3: Layers 2-174 (smallest, top)

**Layer 175: First Merge (#2 + #3)**
1. Detect overlap between #2 and #3
2. Mark #2 and #3 as complete at layer 174
3. Create placeholder #4 starting at layer 175
4. Store pre-merge labels showing which pixels belonged to #2 vs #3
5. Track overlaps for layers 175-204

**Layer 205: Evaluate First Merge**
1. Overlap counts: #2 = 177,300 pixels, #3 = 137,073 pixels
2. **Parent: #2** (more overlap)
3. Update: #3.parentId = 2, #4.parentId = 2, #2.childIds = [3, 4]

**Layer 250: Second Merge (#1 + #4)**
1. Detect overlap between #1 and placeholder #4
2. Resolve #4 to its true parent #2
3. Now merging #1 and #2 (resolved from #4)
4. Mark #1 and #2 as complete at layer 249
5. Create placeholder #5 starting at layer 250
6. Track overlaps for layers 250-279

**Layer 280: Evaluate Second Merge**
1. Overlap counts: #1 = 362,430 pixels, #2 = 362,520 pixels (very close!)
2. **Parent: #1** (slightly more overlap)
3. Update: #2.parentId = 1, #5.parentId = 1, #1.childIds = [2, 5]

**Final Result:**
- Island #1: 505.99 mm³, parent of #2
- Island #2: 173.97 mm³, child of #1, parent of #3
- Island #3: 174.01 mm³, child of #2
- Placeholders #4 and #5 filtered out

### Key Implementation Files

- **`islandTracker.ts`**: Core logic for island ID propagation, merge detection, and pending merge tracking
- **`ScanOrchestrator.ts`**: Orchestrates scanning, calculates volumes, filters placeholders
- **`types.ts`**: Island interface with `perLayerAreaMm2`, `isMergedPlaceholder`, parent/child IDs

### Performance Considerations

- **Memory**: Storing pre-merge labels adds ~4 bytes per pixel per pending merge
- **Evaluation Window**: 30 layers is a balance between accuracy and memory usage
- **Complexity**: O(n) for overlap tracking per layer, O(1) for parent determination

### Edge Cases Handled

1. **Identical cross-sections**: Resolved by overlap tracking over 30 layers
2. **Multiple simultaneous merges**: Each tracked independently with separate pending merges
3. **Nested placeholders**: Resolved recursively to find true parent
4. **Islands that never merge**: Remain as standalone islands with no parent

---

## Summary

The island volume and hierarchy system uses:
1. **Pre-merge area maps** for accurate individual volume calculations
2. **30-layer evaluation windows** for determining true parents based on overlap
3. **Placeholder resolution** to ensure hierarchies reference real islands only
4. **Filtering** to present only meaningful islands to the user

This approach provides accurate, intuitive results even for complex geometries with identical cross-sections at merge points.
