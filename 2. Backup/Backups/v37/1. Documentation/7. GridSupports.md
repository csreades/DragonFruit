# Grid Support System

## Overview

The Grid Support system is designed to create a structured, organized network of supports rather than placing them arbitrarily. Imagine a checkerboard or grid projected onto the build plate. When you place a support, instead of it landing exactly where you clicked, it snaps to the nearest intersection point on this invisible grid.

This system has two main benefits:
1.  **Organization:** It keeps the support structure clean and predictable.
2.  **Efficiency:** It allows multiple support tips to share a single "trunk" or pillar. If a new support point is close to an existing support, it can "branch" off that existing support instead of creating a whole new pillar from the ground. This saves material and print time.

### Key Concepts

#### 1. Trunks
A **Trunk** is the main vertical pillar of a support structure. It starts from the build plate (or a raft) and extends upwards. In the grid system, trunks are always positioned at the exact intersection points of the grid (defined by the grid spacing). A trunk serves as the "backbone" that can hold up multiple branches.

#### 2. Branches
A **Branch** is a smaller support arm that extends from a trunk to touch the model. Instead of going all the way to the ground, a branch connects to a trunk.
*   **Logic:** When you add a support in grid mode, the system checks if there's already a trunk nearby (at the grid point).
*   **Connection:** If a trunk exists, the new support becomes a branch. It calculates the best path to connect to the trunk, ensuring it's not too steep (respecting a minimum angle) and that it snaps cleanly to the trunk's side.

#### 3. Grid Snapping
**Grid Snapping** is the mechanism that forces supports to align. Even if you click slightly off-center, the system calculates the nearest grid node and places the support's base there. This ensures that if you click multiple times in the same general area, all those supports will try to share the same trunk location.

## Technical Implementation

The grid system is built on top of the existing `SupportInstance` data structure but introduces specific logic for placement and merging.

### Data Structures

**`SupportGridSettings`**
This interface defines the configuration for the grid:
*   `enabled`: Boolean flag to toggle the system.
*   `spacingMm`: The distance between grid nodes (e.g., 4mm).

This is integrated into the global `SupportSettings`.

### Core Logic

#### 1. Grid Snapping (`snapToGrid`)
Located in `src/supports/Grid/merging.ts`.
This utility takes a raw coordinate (x, y) and rounds it to the nearest multiple of the `spacingMm`.
```typescript
x = Math.round(x / spacing) * spacing
y = Math.round(y / spacing) * spacing
```

#### 2. Merging Logic (`calculateGridMerge`)
Located in `src/supports/Grid/merging.ts`.
This is the brain of the system. When a new support is proposed:
1.  It calculates the snapped base position.
2.  It searches the existing `supports` list for a support that already occupies that base position (within a small tolerance).
#### 3. Trunk Selection Logic
When multiple supports share the same grid node (including the new candidate), the system determines which one acts as the "Trunk" based on height:
*   **Tallest Support Wins:** The system compares the tip height (Z-value) of the new support against the existing trunk.
*   **New Trunk:** If the **new support is taller** (tip is higher), it becomes the new Trunk. The existing support is converted into a branch connecting to this new, taller trunk.
*   **Branch:** If the **new support is shorter** (tip is lower), the existing trunk remains the Trunk, and the new support becomes a branch connecting to it.
*   **Grid Node Priority:** This comparison only happens if the supports are snapped to the *same* grid node. If they are on different nodes, they remain separate trunks.

#### 4. Branch Snapping & Geometry
The system ensures branches connect physically to the trunk geometry:
*   It uses `snapToTrunk` to find the surface of the trunk shaft.
*   It respects `MIN_BRANCH_ANGLE_DEG` (45 degrees) to ensure printability.
*   If a branch would be too steep, it adjusts the connection point downwards on the trunk.

### Integration

**`useTrunkPlacement` Hook**
The grid logic is hooked into the user interaction layer here.
*   **Preview:** When hovering, the hook calls `calculateGridMerge` to show the user exactly what will happen (trunk or branch) before they click.
*   **Placement:** On click, the same logic is applied to commit the support to the store.
*   **Updates:** The hook subscribes to store updates to ensure the preview refreshes immediately if settings (like grid spacing or presets) change.

**Presets**
Presets (Detail, Structure, Anchor) now include grid settings, allowing users to save their preferred grid configuration (enabled/disabled, spacing) alongside their physical support parameters.
